# ESG-методология и анкета для самооценки компаний

**Версия:** 1.0
**Основа:** синтез LSEG ESG Scores Methodology (v1.0, март 2026) как базовой структуры + элементы MSCI (risk/management-подход), S&P Global CSA (disclosure-gated scoring), Sustainalytics (severity-scale controversy overlay) и Fitch (пятиуровневые границы грейдов, негативные флаги активности)

---

## 1. Общая архитектура

Методология построена на четырёх уровнях агрегации, повторяющих логику LSEG, но с гибридными элементами других провайдеров:

```
Вопрос анкеты → Категория → Столп (E/S/G) → Итоговый ESG-скор → Тир зрелости
                                                     ↑
                                          Controversy Overlay (капы)
```

1. **Индикатор (вопрос анкеты)** — атомарный ответ, нормализуется в шкалу 0–100.
2. **Категория** — группа индикаторов (например, "Выбросы", "Управление советом директоров").
3. **Столп (Pillar)** — Environment / Social / Governance, взвешенная сумма категорий.
4. **ESG-скор** — взвешенная сумма трёх столпов, веса зависят от отрасли (industry-weighted, как в LSEG и S&P).
5. **Controversy Overlay** — отдельный слой негативных флагов и контроверсий, который **капает** (ограничивает сверху) итоговый скор, независимо от того, насколько хороши остальные ответы — логика взята из Sustainalytics (severity categories) и Fitch (negative activity flags).

## 2. Категории по столпам

Структура повторяет привычную LSEG-подобную декомпозицию на 10 категорий:

### Environment (E)
| Категория | Что оценивает |
|---|---|
| E1. Ресурсопользование | Энергия, вода, материалы, эффективность использования |
| E2. Выбросы и отходы | GHG-выбросы, загрязнение воздуха, отходы, биоразнообразие |
| E3. Экологические инновации | Зелёная выручка, эко-R&D, зелёные патенты/продукты |

### Social (S)
| Категория | Что оценивает |
|---|---|
| S1. Персонал | Условия труда, охрана труда, обучение, текучесть |
| S2. Права человека и цепочка поставок | Политики, due diligence поставщиков |
| S3. Сообщество | Локальные инвестиции, филантропия, воздействие на сообщества |
| S4. Ответственность за продукт | Качество/безопасность продукта, защита данных, маркетинг |

### Governance (G)
| Категория | Что оценивает |
|---|---|
| G1. Менеджмент | Структура совета директоров, независимость, диверсити |
| G2. Права акционеров | Защита прав акционеров, антизахватные механизмы |
| G3. CSR-стратегия и этика | Комплаенс, антикоррупционная политика, ESG-отчётность |

### Controversy Overlay (отдельно, не входит в веса столпов)
Негативные флаги и контроверсии — см. раздел 6.

## 3. Шкала ответов и нормализация

Чтобы упростить реализацию в JS, все качественные вопросы используют единую 5-точечную шкалу:

| Ответ | Балл |
|---|---|
| Отсутствует / Нет | 0 |
| Начальный уровень / в планах | 25 |
| Частично внедрено | 50 |
| Систематически внедрено | 75 |
| Продвинутый / лидирующий уровень | 100 |

Количественные вопросы (например, "% возобновляемой энергии", "% женщин в совете директоров") нормализуются линейно между нижним порогом (0 баллов) и целевым/лидирующим порогом (100 баллов), который указан в самом вопросе. Это близко к тому, как S&P Global CSA использует бенчмарки внутри отрасли, только упрощённо — под один общий порог, а не под percentile-ранкинг.

## 4. Формулы агрегации

**Шаг 1. Скор категории**
```
CategoryScore_i = среднее взвешенное баллов всех вопросов категории i
```

**Шаг 2. Disclosure Gate (штраф за нераскрытие, логика S&P CSA)**
Если доля вопросов категории, на которые дан содержательный ответ (не "нет данных"), меньше 50% —
применяется штрафной множитель:
```
DisclosureMultiplier_i = 0.8   (если ответили < 50% вопросов категории)
DisclosureMultiplier_i = 1.0   (если ответили ≥ 50% вопросов категории)

CategoryScore_i_adj = CategoryScore_i × DisclosureMultiplier_i
```
Смысл: компания, которая просто не раскрывает данные, не должна получать среднюю оценку "по умолчанию" — это поощряло бы недостаток прозрачности.

**Шаг 3. Скор столпа**
```
PillarScore_E = Σ (CategoryWeight_i × CategoryScore_i_adj), для категорий E1–E3, веса суммируются в 100%
PillarScore_S = аналогично для S1–S4
PillarScore_G = аналогично для G1–G3
```
Внутри столпа веса категорий по умолчанию равны (например, E1=E2=E3=33.3%), но их можно калибровать под конкретную отрасль позже.

**Шаг 4. Индустриально-взвешенный ESG-скор (логика LSEG / S&P)**
```
ESG_raw = w_E × PillarScore_E + w_S × PillarScore_S + w_G × PillarScore_G
```
где w_E, w_S, w_G зависят от отрасли компании (см. таблицу в разделе 5), сумма весов = 100%.

**Шаг 5. Controversy Cap (логика Sustainalytics + Fitch)**
```
ESG_final = min(ESG_raw, ControversyCap)
```
где `ControversyCap` определяется максимальной severity сработавшего негативного флага (раздел 6).

**Шаг 6. Присвоение тира зрелости**
См. таблицу в разделе 7.

## 5. Индустриальные веса столпов (industry-weighted, LSEG/S&P-подход)

Двухуровневая система, ближе к GICS-классификации, которую используют MSCI/S&P:

- **Уровень 1 (обязательный):** пользователь выбирает сектор из выпадающего списка (13 групп + "Прочее").
- **Уровень 2 (опциональный):** если для сектора есть под-отрасли, показывается второй выпадающий список. Если пользователь его пропускает — используются веса сектора по умолчанию (уровень 1). Если выбирает под-отрасль — её веса **полностью заменяют** веса сектора (override, а не модификатор), чтобы логика в коде оставалась простой: одна финальная тройка весов w_E/w_S/w_G, без дополнительной пере-нормализации.

Все веса — иллюстративные стартовые значения, нужно калибровать под реальные данные позже. Сама логика (у "грязных"/капиталоёмких отраслей выше вес E, у финансового сектора и IT — выше вес G) общепринята во всех крупных методологиях.

```
1. Пользователь выбирает Sector (обязательное поле) → weights = SECTOR_WEIGHTS[sector]

3. Если пользователь выбрал SubIndustry → weights = SUBINDUSTRY_WEIGHTS[sector][subIndustry]

4. weights передаются в Шаг 4 формулы агрегации (раздел 4) как w_E, w_S, w_G
```

## 6. Controversy Overlay — негативные флаги

Гибрид из двух источников:
- **Sustainalytics** даёт идею градации по severity (5 категорий тяжести).
- **Fitch** даёт идею жёстких негативных активити-флагов (например, участие в определённых видах деятельности) как отдельного триггера, а не просто вычитаемого балла.

### 6.1. Вопросы-флаги (Да/Нет + при "Да" — уровень тяжести)

| № | Флаг | Категория тяжести при срабатывании |
|---|---|---|
| C1 | Компания вовлечена в производство спорных видов вооружений | Критическая (4) |
| C2 | Более 30% выручки — тепловой уголь / нефтеносные пески | Тяжёлая (3) |
| C3 | Доказанные системные нарушения прав человека в цепочке поставок | Тяжёлая (3) |
| C4 | Крупные штрафы регулятора за экологические нарушения за последние 3 года | Средняя (2) |
| C5 | Судимость/санкции руководства за коррупцию за последние 5 лет | Тяжёлая (3) |
| C6 | Крупная утечка персональных данных клиентов за последний год | Средняя (2) |
| C7 | Массовые трудовые споры / забастовки, связанные с условиями труда | Лёгкая (1) |
| C8 | Отсутствие какой-либо экологической политики при высокой материальности E для отрасли | Лёгкая (1) |

### 6.2. Логика капа

Берётся **максимальная** сработавшая severity среди всех флагов:

| Severity | Cap на ESG_final | Дополнительно |
|---|---|---|
| 0 (нет флагов) | без ограничений | — |
| 1 (лёгкая) | максимум 90 | — |
| 2 (средняя) | максимум 75 | — |
| 3 (тяжёлая) | максимум 55 | — |
| 4 (критическая) | максимум 35 | тир не может быть выше "Базовый/Отстающий" (см. раздел 7), независимо от числового скора |

## 7. Итоговая шкала тиров зрелости (5-уровневая, Fitch-style границы)

| Диапазон ESG_final | Тир | Обозначение |
|---|---|---|
| 85–100 | Тир 1 | Лидирующий (аналог AAA/AA) |
| 70–84 | Тир 2 | Продвинутый (аналог A/BBB) |
| 55–69 | Тир 3 | Развивающийся (аналог BBB-/BB) |
| 40–54 | Тир 4 | Базовый (аналог B/CCC) |
| 0–39 | Тир 5 | Отстающий / высокий риск (аналог CCC-/D) |

## 8. Анкета

Ниже — базовый набор вопросов по каждой категории. Для каждого вопроса указан тип ответа и что он означает в баллах. Этот набор — стартовый костяк; при необходимости можно расширять количество вопросов внутри категорий без изменения формул агрегации.

### E1. Ресурсопользование
1. Есть ли у компании формализованная политика энергоэффективности? *(шкала 0–100)*
2. Какая доля потребляемой энергии приходится на возобновляемые источники? *(0% → 0 баллов, 100% → 100 баллов, линейно)*
3. Ведётся ли систематический учёт водопотребления? *(шкала 0–100)*
4. Есть ли программа снижения материалоёмкости производства/услуг? *(шкала 0–100)*

### E2. Выбросы и отходы
5. Ведётся ли учёт выбросов парниковых газов (Scope 1 и 2)? *(шкала 0–100)*
6. Есть ли у компании публично заявленная цель по снижению выбросов? *(шкала 0–100)*
7. Есть ли программа управления отходами (переработка, снижение отходов)? *(шкала 0–100)*
8. Проводится ли оценка воздействия деятельности на биоразнообразие? *(шкала 0–100)*

### E3. Экологические инновации
9. Какая доля выручки приходится на "зелёные" продукты/услуги? *(0% → 0, ≥30% → 100, линейно)*
10. Инвестирует ли компания в R&D, направленный на снижение экологического воздействия? *(шкала 0–100)*

### S1. Персонал
11. Есть ли формализованная система охраны труда и техники безопасности? *(шкала 0–100)*
12. Какой процент сотрудников проходит регулярное профессиональное обучение? *(0% → 0, ≥80% → 100, линейно)*
13. Публикует ли компания данные по текучести кадров? *(шкала 0–100)*
14. Есть ли политика по гендерному/диверсити балансу в найме? *(шкала 0–100)*

### S2. Права человека и цепочка поставок
15. Есть ли формализованная политика по правам человека? *(шкала 0–100)*
16. Проводится ли due diligence поставщиков на предмет трудовых/экологических практик? *(шкала 0–100)*
17. Есть ли механизм подачи жалоб для сотрудников/поставщиков (whistleblowing)? *(шкала 0–100)*

### S3. Сообщество
18. Инвестирует ли компания в локальные сообщества (программы, гранты, партнёрства)? *(шкала 0–100)*
19. Оценивается ли социальное воздействие деятельности компании на местные сообщества? *(шкала 0–100)*

### S4. Ответственность за продукт
20. Есть ли формализованная система контроля качества и безопасности продукции? *(шкала 0–100)*
21. Есть ли политика защиты персональных данных клиентов? *(шкала 0–100)*
22. Проводится ли проверка маркетинговых материалов на предмет вводящих в заблуждение заявлений? *(шкала 0–100)*

### G1. Менеджмент
23. Какая доля независимых директоров в составе совета? *(0% → 0, ≥50% → 100, линейно)*
24. Есть ли комитет по устойчивому развитию/ESG на уровне совета директоров? *(шкала 0–100)*
25. Какая доля женщин в составе совета директоров? *(0% → 0, ≥40% → 100, линейно)*

### G2. Права акционеров
26. Обеспечено ли равное право голоса для всех акционеров (один голос — одна акция)? *(шкала 0–100)*
27. Есть ли прозрачная политика вознаграждения руководства, привязанная к ESG-показателям? *(шкала 0–100)*

### G3. CSR-стратегия и этика
28. Есть ли формализованная антикоррупционная политика? *(шкала 0–100)*
29. Публикует ли компания ежегодный ESG/устойчивый отчёт? *(шкала 0–100)*
30. Есть ли независимая верификация ESG-отчётности третьей стороной? *(шкала 0–100)*

### Controversy Overlay (см. раздел 6.1)
31–38. Восемь флаг-вопросов C1–C8 (Да/Нет + при "Да" уточнение степени тяжести, если применимо).

## 9. Дизайн роадмапа рекомендаций

**Принципиальное решение: роадмап — полностью rule-based (без LLM/ИИ-генерации).**
Причины: воспроизводимость (одинаковые ответы → одинаковый роадмап), объяснимость (каждая рекомендация трассируется до конкретного правила и балла — важно для защиты проекта), отсутствие затрат/задержек на внешний API в MVP, и гарантия, что содержание рекомендаций соответствует именно нашей методологии, а не додумано моделью. ИИ-полировка текста (перефразирование в связный абзац без изменения содержания) остаётся опциональным улучшением на будущее, не частью MVP.

### 9.1. Три источника рекомендаций

Роадмап собирается не из одного правила "низкий балл → совет", а из трёх разных сигналов с разным тоном:

1. **Disclosure gaps** (`disclosure_flags[cat] == true`) — компания, возможно, уже что-то делает, просто не раскрыла данные в анкете. Тон: "начните систематически собирать/публиковать данные", а не "внедрите с нуля". Горизонт всегда "быстро".
2. **Низкие категориальные баллы** (`category_score < 60`, без disclosure-проблемы) — структурный пробел, компания реально не делает того, что спрашивается. Тон: конкретная content-рекомендация, что внедрить.
3. **Сработавшие controversy-флаги** — отдельный, более тревожный блок, не смешивается с обычными рекомендациями. Тон: "прямой риск для рейтинга, требует немедленного внимания", а не "было бы неплохо улучшить".

### 9.2. Приоритизация — по impact, не по алфавиту

Используются уже посчитанные industry-weights столпов. Формула приоритета:
```
priority_score = pillar_weight × (100 − category_score)
```
Чем выше `priority_score` — тем выше рекомендация в списке. Логика: низкий балл в столпе с высоким весом для отрасли (например, низкий E у нефтегазовой компании, где w_E = 60%) важнее, чем такой же низкий балл в менее материальном для отрасли столпе.

### 9.3. Горизонт внедрения (quick / mid / long)

Каждой категории присвоен горизонт по умолчанию (сложность/скорость внедрения), адаптировано из логики BC ESG:

| Категория | Горизонт | Обоснование |
|---|---|---|
| E1 Ресурсопользование | Средний | программы эффективности требуют времени на внедрение |
| E2 Выбросы и отходы | Долгий | измеримые цели по выбросам — многолетний трек |
| E3 Экологические инновации | Долгий | R&D и новые продукты — структурные изменения |
| S1 Персонал | Средний | программы обучения/условий труда |
| S2 Права человека / цепочка поставок | Средний | due diligence поставщиков требует времени на выстраивание |
| S3 Сообщество | Быстрый | программы вовлечения можно запустить относительно быстро |
| S4 Ответственность за продукт | Средний | системы контроля качества/данных |
| G1 Менеджмент | Долгий | изменение состава совета директоров — структурный процесс |
| G2 Права акционеров | Долгий | изменение корпоративного управления |
| G3 CSR-стратегия и этика | Быстрый | формализация политик — относительно быстрый шаг |

**Disclosure gaps всегда переопределяют горизонт на "быстрый"**, независимо от категории — раскрытие уже существующих данных не требует структурных изменений.

### 9.4. Шаблоны текста по диапазону балла

Для каждой категории — библиотека из 3 текстовых шаблонов, в зависимости от диапазона балла (не общая фраза "улучшите G1"):
- **0–25** — "практика отсутствует полностью" → рекомендация с нуля
- **25–50** — "начальный уровень / в планах" → рекомендация по формализации и систематизации
- **50–75** (но с фактическим триггером только при score < 60) — "частично внедрено" → рекомендация по усилению/детализации существующей практики

### 9.5. Итоговая структура отчёта

```
1. Общий скор + тир + одна фраза-вывод
2. Разбивка по столпам (E/S/G) — визуально, что тянет вниз
3. ⚠️ Controversy-блок (если есть сработавшие флаги) — отдельно, наверху, как риск
4. Топ-3 приоритетных действия (по priority_score) — "если сделать только одно — сделайте это"
5. Полный список рекомендаций, сгруппированный по горизонту (быстро / средне / долго)
```

## 10. Источники синтеза

- LSEG ESG Scores Methodology (v1.0, март 2026) — базовая структура категорий и столпов, industry-weighted подход
- MSCI ESG Ratings — идея key-issues по отраслям (37 issues / 10 тем), rules-based industry-relative буквенная шкала
- S&P Global CSA — disclosure-gated подход (штраф за нераскрытие), double materiality, процентильная шкала внутри отрасли
- Sustainalytics ESG Risk Ratings — exposure/management фреймворк, 5-категорийная шкала severity для событий-контроверсий
- Sustainable Fitch — формализованные границы 5-уровневых грейдов, негативные activity-флаги как отдельный триггер (не просто вычитание баллов)
